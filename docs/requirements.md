# Документация по Telegram-боту для обработки заявок и предложений по улучшению

## 1. Назначение системы

Telegram-бот предназначен для приема обращений от пользователей. Бот должен поддерживать два основных типа заявок:

1. **Appeal** — жалоба / обращение.
2. **Improvement** — предложение по улучшению.

На текущем этапе бот должен:

- приветствовать нового пользователя;
- запрашивать обязательные данные для идентификации;
- давать пользователю возможность выбрать язык;
- принимать ИНН;
- принимать текст и медиа в рамках заявки;
- после успешной отправки показывать номер заявки, статус и сообщение о дальнейшем контакте.

На будущее нужно учесть расширение, при котором пользователь сможет проверять статус своей заявки через бот, но этот сценарий сейчас не реализуется.

---

## 2. Общий пользовательский сценарий

Логика работы бота соответствует следующему потоку:

**Старт → выбор языка → ввод контактных данных / идентификация → ввод ИНН → выбор типа обращения → для Appeal: выбор домена → при наличии: выбор сути проблемы → отправка текста и/или медиа → регистрация заявки → показ номера заявки и статуса**

Также в будущем из точки после идентификации должен быть доступен отдельный сценарий:

**Проверка статуса заявки по номеру**

---

## 3. Поддерживаемые сценарии

## 3.1. Авторизация / первичный вход

### Описание

При первом входе в бот пользователь должен пройти обязательную начальную инициализацию.

### Обязательные шаги

1. Пользователю показывается приветственное сообщение.
2. Пользователь обязан выбрать язык интерфейса.
3. Пользователь обязан отправить свой ИНН.
4. После успешного прохождения этих шагов пользователь получает доступ к созданию заявки.

### Поддерживаемые языки

- RU
- UZ
- EN

### Требования

- Без выбора языка пользователь не может перейти дальше.
- Без отправки ИНН пользователь не может перейти к созданию заявки.
- Выбранный язык должен сохраняться в профиле пользователя.
- ИНН должен сохраняться в профиле пользователя и использоваться дальше при создании обращений.

---

## 3.2. Создание заявки

### Описание

После авторизации пользователь может открыть одну из двух категорий заявок:

- **appeal**
- **improvement**

Для категории `appeal` сценарий должен быть разветвленным:

1. пользователь выбирает тип обращения `appeal`;
2. бот показывает кнопки выбора домена;
3. если у выбранного домена есть подкатегории по сути проблемы, бот показывает следующий уровень кнопок;
4. после завершения выбора пользователь переходит к шагу описания проблемы и может отправить текст, медиа или оба варианта.

Для категории `improvement` остается прямой сценарий без выбора домена: пользователь сразу переходит к вводу содержимого заявки.

### Возможности

Пользователь может отправить:

- только текст;
- только медиа;
- текст и медиа вместе.

### Результат

После успешной отправки бот должен показать:

- номер заявки;
- статус заявки;
- текст: **«Скоро с вами свяжутся»**.

---

## 3.3. Будущий сценарий: отслеживание статуса заявки

### Описание

На будущем этапе пользователь сможет отправить текстовый запрос на проверку статуса заявки.

### Что нужно учесть в архитектуре уже сейчас

- у каждой заявки должен быть уникальный номер;
- у заявки должен быть статус;
- заявка должна быть связана с пользователем;
- должна существовать возможность получить заявку по номеру и пользователю;
- структура статусов должна быть стандартизована.

### Сейчас не реализуется

На текущем этапе этот сценарий нужно только учесть в проектировании модели данных и логики, но не выводить пользователю в интерфейс как готовую функцию.

# 4. Подробное описание бизнес-логики

## 4.1. Сценарий первичного входа

### Шаг 1. Команда старта

Пользователь заходит в бот и инициирует диалог, например через `/start`.

### Ожидаемое поведение бота

Бот показывает приветственный текст. Пример:

**RU:**
Здравствуйте! Добро пожаловать в бот для подачи обращений и предложений. Для продолжения выберите язык.

**UZ:**
Assalomu alaykum! Murojaat va taklif yuborish botiga xush kelibsiz. Davom etish uchun tilni tanlang.

**EN:**
Hello! Welcome to the bot for submitting appeals and improvement suggestions. Please select a language to continue.

### Шаг 2. Выбор языка

Бот показывает кнопки:

- RU
- UZ
- EN

### Требования к сценарию

- Язык обязателен.
- До выбора языка остальные действия недоступны.
- При повторном заходе выбранный язык не нужно спрашивать заново, если он уже сохранен, если только не предусмотрена отдельная функция смены языка.

### Шаг 3. Запрос ИНН

После выбора языка бот просит пользователя отправить ИНН.

Пример для RU:
**Пожалуйста, отправьте ваш ИНН.**

### Валидация ИНН

Точные правила валидации зависят от вашей предметной области, но в документации стоит закрепить минимум:

- поле обязательно;
- ИНН должен быть строкой;
- допускаются только цифры;
- длина должна соответствовать принятому формату системы.

Если точный формат пока не определен, это нужно явно отметить как открытый вопрос.

### Ошибки

Если ИНН пустой, неверного формата или содержит недопустимые символы, бот должен сообщить об ошибке и повторно запросить ИНН.

Пример:
**ИНН введен некорректно. Пожалуйста, проверьте данные и отправьте снова.**

### Результат успешного шага

После успешного получения ИНН пользователь считается идентифицированным и может перейти к выбору типа заявки.

## 4.2. Сценарий создания заявки

### Шаг 1. Выбор типа обращения

После идентификации бот предлагает выбрать тип заявки:

- **Appeal**
- **Improvement**

Для русской локализации желательно показывать человекочитаемые названия, а внутренне хранить системные коды:

- `appeal` — жалоба / обращение
- `improvement` — предложение по улучшению

### Требования

- Пользователь обязан выбрать один из двух вариантов.
- Пока выбор не сделан, переход к следующему шагу невозможен.

### Шаг 2. Выбор домена для жалобы (`appeal`)

Если пользователь выбрал `appeal`, бот обязан показать кнопки выбора домена проблемы.

Список доменов:

- `Auth` — Регистрация-Вход
- `Transaction` — Оплата
- `Account` — Счета
- `Credit` — Кредиты
- `Reports` — Отчеты
- `Monitoring` — Мониторинг
- `Other` — Другое

### Требования к выбору домена

- выбор домена обязателен только для сценария `appeal`;
- пользователь не может перейти к следующему шагу, пока не выбран домен;
- выбранный домен должен сохраняться в данных заявки;
- пользователю должны показываться кнопки, а не свободный ввод.

### Шаг 3. Выбор сути проблемы внутри домена

Если у выбранного домена есть разветвления, бот должен показать следующий набор кнопок с сутью проблемы.

Поддерживаемые разветвления:

- `Auth`: Подтверждение eimzo, MyID, Забыл пинкод, MFO, Сменился директор, Добавление компании/сотрудника
- `Transaction`: Платежное поручение, Между своими счетами, Система быстрых платежей 24/7, Бюджетные платежи, Шаблоны платежей, Выдача зарплаты, Корпоративные карты
- `Account`: Не вижу счет
- `Credit`: График погашения, Оплата кредита, Кредитные счета

Для доменов `Reports`, `Monitoring` и `Other` отдельный шаг выбора сути проблемы не требуется, если для них не будет добавлена отдельная детализация позже.

### Требования к выбору сути проблемы

- если у домена есть подкатегории, выбор обязателен;
- пользователь не может перейти к описанию проблемы, пока не выбрана подкатегория;
- выбранная подкатегория должна сохраняться в данных заявки;
- если у домена нет подкатегорий, бот сразу переводит пользователя к следующему шагу.

### Шаг 4. Ввод содержимого заявки

После выбора типа заявки и, если это `appeal`, после выбора домена и подкатегории, бот предлагает отправить содержимое заявки.

Пользователь может отправить:

- текст;
- фото;
- видео;
- документ;
- аудио/голосовое сообщение, если это допускается бизнес-требованиями;
- комбинацию текста и медиа.

### Рекомендуемое сообщение

**Опишите вашу проблему. Вы можете отправить текст, медиафайлы или оба варианта вместе.**

### Поддержка вложений

Нужно заранее определить, какие типы файлов допустимы. Базово рекомендуется поддержать:

- image
- video
- document

Если планируется хранить только ссылки на Telegram file_id, это тоже нужно зафиксировать.

### Ограничения

Нужно предусмотреть следующие правила:

- заявка не должна быть пустой;
- должен быть хотя бы один полезный контент: текст или файл;
- при необходимости ограничить число вложений;
- при необходимости ограничить размер файлов.

Если пользователь пытается отправить пустую заявку, бот должен вернуть сообщение об ошибке.

Пример:
**Заявка не может быть пустой. Отправьте текст, медиафайл или оба варианта.**

Для сценария `improvement` этот шаг вызывается сразу после выбора типа обращения.

### Шаг 5. Подтверждение регистрации заявки

После успешного приема данных бот создает заявку в системе и возвращает пользователю подтверждение.

Пользователю обязательно показываются:

- номер заявки;
- статус заявки;
- текст: **«Скоро с вами свяжутся»**

Пример ответа:

```text
Ваша заявка принята.
Номер заявки: 123456
Статус: NEW
Скоро с вами свяжутся.
```

# 5. Функциональные требования

## 5.1. Требования к авторизации

Система должна:

- показывать приветствие при первом входе;
- требовать обязательный выбор языка;
- поддерживать 3 языка: RU, UZ, EN;
- требовать обязательную отправку ИНН;
- сохранять язык и ИНН пользователя.

## 5.2. Требования к заявкам

Система должна:

- позволять открыть один из двух типов заявки: `appeal` или `improvement`;
- для `appeal` обязательно запрашивать домен проблемы;
- для доменов с разветвлениями обязательно запрашивать подкатегорию проблемы;
- для `improvement` сохранять прямой сценарий без выбора домена;
- позволять отправлять текст;
- позволять отправлять медиа;
- позволять отправлять текст и медиа одновременно;
- сохранять выбранный домен и подкатегорию проблемы в данных заявки;
- присваивать каждой заявке уникальный номер;
- устанавливать стартовый статус заявки;
- показывать пользователю подтверждение после успешной регистрации.

## 5.3. Требования к статусу

Для текущего этапа:

- статус должен создаваться автоматически при регистрации заявки;
- статус должен храниться в системе;
- статус должен отображаться пользователю при успешной отправке заявки.

Для будущего этапа:

- статус должен быть доступен для запроса по номеру заявки.

# 6. Нефункциональные требования

## 6.1. Удобство использования

- Диалог должен быть понятным и пошаговым, включая разветвления для сценария `appeal`.
- На каждом шаге пользователь должен понимать, что от него требуется.
- Ошибки должны возвращаться в понятной форме на выбранном языке.

## 6.2. Локализация

Все сообщения бота должны поддерживать RU, UZ и EN.

Минимум локализовать:

- приветствие;
- запрос языка;
- запрос ИНН;
- ошибки валидации;
- выбор типа заявки;
- подтверждение регистрации заявки.

## 6.3. Надежность

- Пользовательские данные не должны теряться между шагами.
- Если произошла ошибка при сохранении заявки, пользователь должен получить понятное уведомление.
- Повторная отправка одного и того же сообщения не должна создавать дубли без необходимости.

## 6.4. Безопасность

- ИНН является чувствительными данными, его нужно хранить и передавать безопасно.
- Желательно логировать системные ошибки без утечки персональных данных в логи.
- Нужно разграничить пользовательские данные и служебные данные заявки.

# 7. Состояния пользователя

Для реализации бота удобно ввести конечный автомат состояний.

## Возможные состояния

- `NEW_USER` — новый пользователь
- `WAIT_LANGUAGE` — ожидание выбора языка
- `WAIT_INN` — ожидание ИНН
- `READY_FOR_REQUEST_TYPE` — пользователь может выбрать тип заявки
- `WAIT_REQUEST_CONTENT` — ожидание текста/медиа заявки
- `REQUEST_CREATED` — заявка создана

## Переходы

1. Новый пользователь входит в бот → `WAIT_LANGUAGE`
2. Выбран язык → `WAIT_INN`
3. ИНН успешно получен → `READY_FOR_REQUEST_TYPE`
4. Выбран тип заявки → `WAIT_REQUEST_CONTENT`
5. Контент заявки успешно получен → `REQUEST_CREATED`
6. После создания заявки пользователь может снова вернуться к `READY_FOR_REQUEST_TYPE`

Это позволит нормально реализовать FSM и не путаться в многошаговом диалоге.

# 8. Состояния заявки

Рекомендуется уже сейчас заложить набор статусов, даже если часть понадобится позже.

Минимально:

- `NEW` — заявка создана
- `IN_PROGRESS` — в работе
- `RESOLVED` — обработана
- `REJECTED` — отклонена
- `CLOSED` — закрыта

На текущем этапе после создания новой заявки ей присваивается статус:

- `NEW`

Именно этот статус показывается пользователю после отправки.

# 9. Модель данных

## 9.1. Пользователь

### Поля

- `id` — внутренний идентификатор
- `telegram_user_id` — идентификатор пользователя Telegram
- `telegram_username` — username Telegram, если есть
- `full_name` — имя пользователя, если нужно
- `language` — выбранный язык: RU / UZ / EN
- `inn` — ИНН пользователя
- `created_at`
- `updated_at`

## 9.2. Заявка

### Поля

- `id` — внутренний идентификатор
- `request_number` — публичный номер заявки
- `user_id` — ссылка на пользователя
- `type` — `appeal` или `improvement`
- `text` — текст заявки, если есть
- `status` — текущий статус
- `created_at`
- `updated_at`

## 9.3. Вложения заявки

Если медиа нужно хранить отдельно, лучше вынести в отдельную сущность.

### Поля

- `id`
- `request_id`
- `telegram_file_id`
- `file_type`
- `file_name`
- `mime_type`
- `file_size`
- `created_at`

Такое разделение нормальное и масштабируемое. Не стоит хранить вложения прямо одной строкой внутри заявки, если файлов может быть несколько.

# 10. API/логика уровня приложения

Если бот общается с backend API, то логика может быть такой.

## 10.1. Создание или получение пользователя

При `/start`:

Отдельный метод:

- принимает `language`

## 10.3. Сохранение ИНН

Отдельный метод:

- принимает `telegram_user_id`
- принимает `inn`

## 10.4. Создание заявки

Метод должен принимать:

- `telegram_user_id` или `user_id`
- `type`
- `text`
- массив вложений
- язык пользователя при необходимости

Возвращает:

- номер заявки
- статус

# 12. Валидации и проверки

## Для языка

- значение должно быть только из списка: `RU`, `UZ`, `EN`

## Для ИНН

- обязательное поле
- только допустимые символы
- длина по утвержденному формату

## Для заявки

- обязателен тип заявки
- должна быть хотя бы одна полезная единица контента:
  - текст, или
  - файл, или
  - текст + файл

## Для вложений

- проверка допустимых типов
- проверка размеров
- ограничение количества вложений

# 13. Ошибочные сценарии

## Сценарий 1. Пользователь не выбрал язык

Бот не должен пускать дальше и должен повторно показать кнопки выбора языка.

## Сценарий 2. Пользователь отправил некорректный ИНН

Бот должен показать ошибку и снова ждать ИНН.

## Сценарий 3. Пользователь не выбрал тип заявки, но пытается отправить контент

Бот должен вернуть пользователя на шаг выбора типа заявки.

## Сценарий 4. Пользователь отправил пустое сообщение

Бот должен сообщить, что заявка не может быть пустой.

## Сценарий 5. Сбой при сохранении заявки

Бот должен показать нейтральное сообщение:
**Произошла ошибка при отправке заявки. Пожалуйста, попробуйте позже.**

# 14. Требования для будущего расширения

Хотя проверка статуса сейчас не реализуется, уже сейчас рекомендуется заложить:

1. Уникальный номер заявки, удобный для пользовательского ввода.
2. Возможность искать заявку по номеру.
3. Привязку заявки к конкретному пользователю.
4. Таблицу или поле для истории изменения статусов.
5. Стандартизованный набор статусов.

### Будущий сценарий

Пользователь пишет номер заявки или специальную команду, а бот возвращает:

- номер заявки;
- текущий статус;
- дату создания;
- возможно, возможность отмены/удаления заявки, если это потом понадобится.

# 15. Критичные вопросы, которые лучше утвердить заранее

Есть несколько моментов, которые лучше не оставлять “на потом”, иначе потом придется переделывать схему и код:

## 1. Формат ИНН

Нужно точно определить:

- сколько символов;
- только цифры или нет;
- нужна ли проверка контрольной суммы.

## 2. Контактные данные

На схеме есть блок **контактные данные**, но в тексте требований указан только ИНН.
Нужно определить, действительно ли нужны дополнительные контакты, например:

- ФИО
- телефон
- email

Или же Telegram + ИНН уже достаточно.

## 3. Формат номера заявки

Нужно решить, каким будет номер:

- автоинкремент;
- UUID;
- человекочитаемый номер, например `REQ-20260402-00125`.

Для пользователя лучше человекочитаемый формат.

# 17. Краткое итоговое ТЗ

Бот должен:

1. При первом входе показать приветствие.
2. Обязать пользователя выбрать язык: RU / UZ / EN.
3. Обязать пользователя отправить ИНН.
4. После этого дать выбрать тип заявки: `appeal` или `improvement`.
5. Принять от пользователя текст, медиа или оба варианта.
6. Создать заявку в системе.
7. Показать:
   - номер заявки,
   - статус,
   - сообщение: **«Скоро с вами свяжутся»**.
8. На уровне архитектуры предусмотреть будущую возможность проверки статуса заявки по текстовому запросу.
