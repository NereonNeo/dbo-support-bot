Ниже детальный план реализации на основе требований и текущей архитектуры проекта.

**Цели итерации**

1. Реализовать линейный сценарий: старт → язык → ИНН → тип обращения → контент → создание заявки → подтверждение.
2. Обеспечить хранение пользователей, заявок и вложений в Postgres через Prisma.
3. Заложить основу под будущий сценарий проверки статуса по номеру, без вывода в UI.

**Этап 1. Подготовка и базовая инфраструктура**

1. Проверить конфигурацию окружения.
2. Уточнить `.env` переменные: `BOT_ID`, `DATABASE_URL`.
3. Подтвердить формат ИНН и правила валидации.
4. Определить формат `request_number`.
5. Подтвердить набор допустимых типов вложений и ограничения по размеру и количеству.

**Этап 2. Модель данных и Prisma**

1. Расширить модель `User` полями `inn`, `createdAt`, `updatedAt`.
2. Добавить модель `Request` со связью на `User`.
3. Добавить модель `RequestAttachment` со связью на `Request`.
4. Зафиксировать enum'ы для `RequestType` и `RequestStatus`.
5. Добавить уникальный `requestNumber`.
6. Сгенерировать миграцию и обновить Prisma клиент.

**Этап 3. Состояния и FSM**

1. Ввести статусы пользователя и их переходы согласно требованиям.
2. Определить хранение состояния на уровне пользователя, например `user.state`.
3. Добавить обработку состояния в middleware или shared-слое.
4. Обеспечить возврат в `READY_FOR_REQUEST_TYPE` после успешной заявки.

**Этап 4. Локализация**

1. Создать словари RU/UZ/EN для ключевых сообщений.
2. Перекрыть текущие hardcoded сообщения в `auth` на локализованные.
3. Можно использовать i18n,

**Этап 5. Авторизация и идентификация**

1. `auth.handler`: при `/start` проверять наличие пользователя и статус.
2. Если язык не выбран, показать клавиатуру языков.
3. После выбора языка запросить ИНН.
4. Валидировать ИНН, сохранять в `User`.
5. Обработать ошибки ИНН с повторным запросом.

**Этап 6. Обработка заявок**

1. Добавить `handlers/requests` домен в структуре как отдельный модуль.
2. Реализовать выбор типа обращения с inline-кнопками.
3. Реализовать сбор текста и медиа:
4. Хранить временные данные заявки до подтверждения в памяти или в БД.
5. Проверить, что заявка не пустая.
6. При успешной отправке создать запись `Request` и `RequestAttachment`.
7. Вернуть пользователю `request_number`, `status`, и текст «Скоро с вами свяжутся».

**Этап 7. Валидации и ограничения**

1. Валидация языка, ИНН, типа заявки.
2. Проверка типов файлов и допустимых размеров.
3. Ограничение количества вложений, если требуется бизнесом.
4. Защитная логика от дублирования заявки при повторной отправке одного и того же сообщения.

**Этап 8. Архитектурные заделы для будущего статуса**

1. Реализовать хранение статуса в `Request` и подготовить индексы.
2. Обеспечить возможность поиска по `request_number` и `user_id`.
3. Не добавлять пользовательский интерфейс для проверки статуса.

**Этап 9. Обработка ошибок и надежность**

1. Унифицировать ответы об ошибках через локализацию.
2. Логировать системные ошибки без утечки персональных данных.
3. Вернуть понятный ответ пользователю при сбое сохранения заявки.

**Этап 10. Тестирование и верификация**

1. Прогнать сценарии happy-path на всех языках.
2. Проверить ошибки: пустая заявка, неверный ИНН, отсутствие типа заявки.
3. Проверить создание заявки с текстом, медиа и комбинированно.
4. Проверить корректность сохранения статуса `NEW`.

**Этап 11. Документация**

1. Обновить `docs/architecture.md` после финальной архитектуры.
2. Зафиксировать решения по ИНН, номерам заявок, типам вложений.
3. Добавить краткое описание FSM и структуры локализации.

**Открытые вопросы (зафиксированные решения)**

1. ИНН: только цифры, длина 9, без контрольной суммы.
2. Номер заявки: человекочитаемый формат.
3. Дополнительные контактные данные не требуются.
4. Допустимые типы контента: текст, фото, видео.
